1. Overview

Phase 1 implements three tightly connected mechanisms in one demonstrable system:

- a dynamic scheduler with a process table of Task Control Blocks;
- a binary semaphore that blocks and wakes tasks through a queue instead of busy waiting;
- a deterministic first-fit memory manager that allocates, writes, frees, and dumps task buffers.

The current Phase 1 solution is not just a console dump. It is a proof-board style ncurses application that keeps the scheduler state, semaphore state, memory-manager state, task state, event log, and transcript evidence tied together in one executable workflow.

3. Scheduler Design

The scheduler owns a dynamic process table implemented as a vector of Task Control Block pointers.

Each TCB stores:

- task_id
- task_name
- task_state
- task_entry
- context_pointer
- dispatch_count
- yield_count
- block_count
- unblock_count
- detail_note
- last_transition

The scheduler exposes the methods required to drive the phase:

- create_task()
- kill_task()
- yield()
- garbage_collect()
- block_task()
- unblock_task()
- dump()

Behavior summary:

- create_task() allocates a new TCB, assigns a unique task ID, and inserts it into the process table.
- yield() performs strict round-robin selection across READY tasks.
- block_task() moves a task into BLOCKED.
- unblock_task() returns a blocked task to READY.
- kill_task() marks a task DEAD without immediately removing it.
- garbage_collect() deletes DEAD tasks and compacts the process table.

The scheduler also records counters, queue summaries, and last-event text so both the UI and transcript can present readable proof of each transition.

4. Semaphore Design

The semaphore manages one binary resource, Printer_Output.

Internal data tracked by the semaphore:

- resource_name
- initial_value
- value
- scheduler pointer
- wait_queue
- owner_task_id
- down_operations
- up_operations
- contention_events
- last_transition

Behavior summary:

- down() acquires the resource if it is free.
- If the resource is busy, down() increments contention, appends the task to the FIFO wait queue, and blocks the current task through the scheduler.
- up() releases the current owner.
- If the queue is not empty, up() wakes the next queued task and transfers ownership to that task.
- dump() prints the resource name, value, owner, queue, operation counters, and last transition.

This behavior directly satisfies the assignment requirement that semaphore contention must be handled through blocking rather than busy waiting.

5. Memory Manager Design

The memory manager models a small deterministic heap used by the Phase 1 demo to track task-owned buffers.

Internal data tracked by the memory manager:

- heap_bytes
- segments
- used_byte_count

Public operations exposed by the memory manager:

- reset()
- allocate()
- free_block()
- write()
- read()
- describe_layout()
- dump()

Behavior summary:

- reset() restores the heap to one free segment.
- allocate() uses a first-fit scan and returns a stable offset address.
- free_block() releases a segment, zeroes the released bytes, and coalesces adjacent free space.
- write() and read() reject out-of-range access outside an allocated segment.
- dump() prints capacity, used/free bytes, allocation count, and the current free/used segment map.

This gives the Phase 1 proof board a concrete memory-lifetime trace rather than relying on implied cleanup alone.

6. Demonstration Scenario

The live proof uses three tasks, one shared resource, and one task buffer per task.

Task_A:

- first claimant;
- reserves a 160-byte work buffer first;
- acquires Printer_Output first;
- yields while still holding the resource;
- later releases the resource, frees its buffer, and becomes DEAD.

Task_B:

- second claimant;
- reserves a 112-byte wait-record buffer;
- attempts to acquire Printer_Output while Task_A still owns it;
- becomes BLOCKED and enters the wait queue;
- wakes after Task_A releases the resource;
- acquires the resource, releases it, frees its buffer, and becomes DEAD.

Task_C:

- third claimant;
- reserves a 96-byte wait-record buffer;
- attempts to acquire Printer_Output after Task_B;
- becomes BLOCKED behind Task_B in FIFO order;
- wakes only after Task_B releases the resource;
- acquires the resource, releases it, frees its buffer, and becomes DEAD.

This sequence proves:

- READY to RUNNING transitions,
- RUNNING to BLOCKED transitions,
- BLOCKED to READY wake-up behavior,
- deterministic heap allocation and release for all three tasks,
- return to one free heap segment at shutdown,
- DEAD state entry,
- post-run garbage collection,
- and FIFO queue discipline in the semaphore.

7. User Interface and Runtime Flow

The terminal front end is implemented in Ultima.cpp using the U2_ui and U2_window wrappers.

The current preferred presentation layout at 140 columns by 62 rows is a full-width stacked proof board:

- Header
- Scheduler Class
- Semaphore Class
- State + Race + Memory
- Task_A
- Task_B
- Task_C
- Output Log + Dumps
- Controls

Current runtime flow:

1. The UI initializes and creates all windows.
2. The initial system state is displayed and recorded.
3. Task buffers are allocated through Mem_mgr as each task begins its proof path.
4. The scheduler dispatches tasks until no active tasks remain.
5. Final state and post-garbage-collection state are shown on the proof board.
6. In single-cycle mode, the completed windows remain visible.
7. The user presses a key to transition from the final UI to the plain-text transcript.
8. The transcript is written to phase1output.txt.

Continuous mode still exists, but it is now opt-in through the command-line flag --continuous.

8. Current Controls

- h refreshes the help/status panel.
- d appends an on-demand scheduler, semaphore, and memory-manager dump to the log window.
- p pauses or resumes the paced demonstration.
- q exits continuous mode after the current cycle.
- mouse clicks are logged when the terminal supports mouse events.

10. Assignment Compliance Notes

This implementation currently satisfies the assignment expectations in the following ways:

- the process table is dynamic;
- the scheduler tracks RUNNING, READY, BLOCKED, and DEAD states;
- the semaphore uses a FIFO queue and does not busy wait;
- the scheduler, semaphore, and memory manager all expose readable dump output;
- the demonstration visibly proves queueing, blocking, wake-up order, buffer lifetime, and cleanup;
- the project includes both a live terminal demonstration and a transcript artifact;
- the UI is protected by a pthread-based mutual-exclusion boundary so screen rendering remains coherent.

11. Current Limitations and Notes

- Some directories contain old object files, repeated binary copies, and debug symbol folders produced during development; they are not the authoritative source files.
- The real source of truth is the current .cpp, .h, build, and launcher files described above.
- The ncurses UI must run in a real terminal surface. IDE debug consoles can still misrepresent the output even when the build itself is correct.
