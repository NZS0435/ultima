ULTIMA 2.0
Phase 1 Test Plan and Output

1. Objective

The objective of this test plan is to prove that the Phase 1 scheduler, semaphore, and memory-manager implementation satisfies the current repository requirements. The tests focus on five main claims:

- the scheduler maintains a dynamic process table;
- tasks transition correctly through READY, RUNNING, BLOCKED, and DEAD;
- the semaphore blocks on contention instead of busy waiting and wakes queued tasks in FIFO order;
- the memory manager allocates deterministic task buffers and returns the heap to one free segment at shutdown;
- the system produces readable evidence through both the ncurses proof board and the plain-text transcript.

2. Build and Run Procedure

Recommended CMake path:

cmake --preset macos-debug
cmake --build --preset macos-debug
~/.ultima2/build/macos-debug/ultima_os

Shared shell launcher:

./ULTIMA/Phase\ 1\ \(Scheduler\)/run_ultima_ui.sh

Transcript-only launcher:

./ULTIMA/Phase\ 1\ \(Scheduler\)/run_ultima_transcript.sh

Behavior expected from the current default build:

- the program runs one presentation cycle;
- the final task and proof windows remain visible on screen;
- the user presses a key to transition from the completed UI to the transcript;
- the transcript is written to phase1output.txt;
- each system snapshot includes scheduler, semaphore, and memory-manager dumps.

3. Test Scenario

The test uses one shared resource, Printer_Output, three tasks, and three task-owned buffers:

- Task_A is the first claimant, should acquire the semaphore first, and should reserve a 160-byte work buffer at heap offset 0.
- Task_B is the second claimant, should block behind Task_A, and should reserve a 112-byte wait-record buffer at heap offset 160.
- Task_C is the third claimant, should block behind Task_B, and should reserve a 96-byte wait-record buffer at heap offset 272.

Expected execution order:

1. Task_A allocates its buffer and acquires Printer_Output.
2. Task_A yields while still holding the resource.
3. Task_B allocates its buffer, calls down(), and becomes BLOCKED.
4. Task_C allocates its buffer, calls down(), and becomes BLOCKED behind Task_B.
5. Task_A calls up(), frees its buffer, and wakes Task_B.
6. Task_B acquires the resource, releases it, frees its buffer, and becomes DEAD.
7. Task_C acquires the resource, releases it, frees its buffer, and becomes DEAD.
8. Garbage collection removes all DEAD tasks from the process table and the heap returns to one free segment.

4. Test Cases

Test Case 1: Initial process table, semaphore state, and heap state

Expected result:

- three tasks are created in READY state;
- scheduler tick starts at 0;
- current task is -1 before first dispatch;
- semaphore value is 1;
- semaphore wait queue is empty;
- memory-manager capacity is 1024 bytes with 0 bytes used and one free segment.

Observed result:

- transcript header shows Tick: 0 and Current Task: -1;
- run queue shows Task_A, Task_B, and Task_C in READY state;
- semaphore dump shows Value: 1 and Wait Queue: [empty];
- memory dump shows Capacity: 1024 bytes, Used: 0 bytes, and one free segment `[0..1023]`.

Test Case 2: Blocking and queue formation

Expected result:

- after Task_A acquires Printer_Output, Task_B and Task_C must become BLOCKED;
- the semaphore wait queue must show Task_B before Task_C;
- semaphore value must remain 0 while the resource is owned;
- the heap must show three used segments followed by one free segment.

Observed result:

- transcript shows:
  [MemMgr] allocated 160 bytes for Task_A at @0
  [MemMgr] allocated 112 bytes for Task_B at @160
  [MemMgr] allocated 96 bytes for Task_C at @272
  [Task_B] transitioned to BLOCKED.
  [Task_C] transitioned to BLOCKED.
- mid-execution dump shows:
  Task_B BLOCKED
  Task_C BLOCKED
  Wait Queue: Task_B(T-2) -> Task_C(T-3)
  Value: 0
  Owner: 1
  Memory Manager Dump with used segments:
  [0..159] USED size=160 label=Task_A_work
  [160..271] USED size=112 label=Task_B_wait_record
  [272..367] USED size=96 label=Task_C_wait_record

Test Case 3: FIFO wake-up order

Expected result:

- Task_A releasing Printer_Output must wake Task_B first;
- Task_C must remain BLOCKED until Task_B releases the resource;
- Task_A freeing its buffer must reduce heap usage before Task_B runs.

Observed result:

- post-release dump shows:
  Last Event: Unblocked Task_B (T-2).
  Wait Queue: Task_C(T-3)
  Owner: 2
- Task_C remains BLOCKED until Task_B finishes.
- transcript shows:
  [MemMgr] freed buffer for Task_A at @0
- post-release memory dump shows Used: 208 bytes and two remaining task-owned segments.

Test Case 4: Heap cleanup and coalescing

Expected result:

- all task-owned buffers must be freed by the time the final dump is reached;
- the memory manager must return to Used: 0 bytes and one free segment;
- no stale allocation labels may remain after cleanup.

Observed result:

- transcript shows:
  [MemMgr] freed buffer for Task_B at @160
  [MemMgr] freed buffer for Task_C at @272
- final and post-garbage-collection memory dumps show:
  Capacity: 1024 bytes
  Used: 0 bytes
  Free: 1024 bytes
  Segments:
    [0..1023] FREE size=1024

Test Case 5: Completion and task cleanup

Expected result:

- all three tasks must reach DEAD;
- semaphore must return to value 1 with no owner and no waiters;
- garbage_collect() must empty the process table.

Observed result:

- final dump before collection shows Task_A, Task_B, and Task_C all in DEAD state;
- semaphore dump shows Value: 1, Owner: -1, and Wait Queue: [empty];
- post-garbage-collection dump shows an empty process table and Run Queue: [empty].

Test Case 6: UI and transcript handoff

Expected result:

- the proof-board UI remains visible after the final state is reached;
- the user presses a key to show the transcript;
- phase1output.txt is written.

Observed result:

- current build is configured for a single-cycle proof-board presentation;
- transcript output is saved in phase1output.txt in the Phase 1 directory.

5. Current Output Evidence

Key lines from the current transcript artifact:

- Scheduler purpose: manage the dynamic process table and choose the next READY task to run.
- Semaphore purpose: prevent a shared Printer_Output race by blocking and queueing contenders.
- Memory manager purpose: provide deterministic first-fit task buffers and show their lifetime in the proof board.
- [MemMgr] allocated 160 bytes for Task_A at @0
- [MemMgr] allocated 112 bytes for Task_B at @160
- [MemMgr] allocated 96 bytes for Task_C at @272
- FLOW: Task_A -> Task_B BLOCKED -> Task_C BLOCKED
- Wait Queue: Task_B(T-2) -> Task_C(T-3)
- Last Event: Unblocked Task_B (T-2).
- Memory Manager Dump
- Garbage collection removed DEAD tasks from the process table.
- System shutting down safely.

6. Conclusion

The observed output matches the current Phase 1 behavior. The process table is dynamic, state transitions are visible and correct, blocking occurs without busy waiting, FIFO wake-up order is preserved, task-buffer allocation and release are explicit and deterministic, and the final transcript provides readable evidence of the complete execution cycle across the scheduler, semaphore, and memory manager.
